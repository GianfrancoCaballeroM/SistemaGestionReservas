El Proyecto y Objetivos

Plataforma MERN para Reservas de Restaurante

  • Objetivo General: Automatizar el flujo de asignación de mesas y control de capacidad en el restaurante "Mezanine Gourmet".
  • Fidelización (CRM): Centralizar la información de clientes, preferencias alimentarias e historial de asistencia.
  • Marketing Segmentado: Reactivar clientes inactivos según su historial histórico de consumo directamente desde el panel.
  • Experiencia de Usuario: Landing page con menú digital interactivo y envío directo de reseñas públicas.

🎯 Valor del Negocio

Pasar de una gestión en papel a una base de datos distribuida en tiempo real.

Permite predecir horas pico de ocupación y automatizar la consistencia de estadísticas de clientes de manera transaccional e inmediata.

Arquitectura del Sistema

Arquitectura en Capas desacoplada orientada a API REST:

  • Frontend (React): Interfaz desacoplada, desplegada de manera estática y comunicada mediante HTTPS.
  • Capa de Enrutamiento & Seguridad: Controla el acceso público y valida tokens JWT mediante middlewares.
  • Controladores: Contienen la lógica de negocio pura y manejan las peticiones HTTP.
  • Modelos Mongoose: Validan la integridad de los datos antes de persistir en MongoDB Atlas.

Diseño de Modelo de Datos (NoSQL)

Diseño Híbrido: Documentos Embebidos vs. Referenciados

  • Relaciones Embebidas (Desnormalizadas):
    • `Cliente` contiene `notas_staff` (Array).
    • `Reserva` contiene `datos_contacto` (Objeto).
    • *Beneficio:* Lecturas ultrarrápidas y sin Joins.
  • Relaciones Referenciadas (Normalizadas):
    • `Reserva` referencia a `Cliente` vía `cliente_id` (ObjectId).
    • `Resena` referencia a `Cliente` vía `cliente_id`.
    • *Beneficio:* Evita superar el límite de 16MB por documento ante el crecimiento continuo.

Estructura Avanzada de la Base de Datos

🌀 Capped Collection

auditoria_logs (Bitácora de seguridad):

  • Colección de tamaño fijo físico (1MB de capacidad, máximo 1000 documentos).
  • Al llenarse, sobrescribe de forma circular los documentos más antiguos (FIFO).
  • Escrituras físicas ultrarrápidas secuenciales directas en disco.

👁️ Vista Nativa (v_reservas_hoy)

Vista virtual sobre la colección 'reservas':

  • Creada automáticamente en la conexión asíncrona mediante un pipeline de agregación.
  • Filtra las reservas con estado "confirmada" cuya fecha corresponde al día actual utilizando variables dinámicas (`$$NOW`).
  • Desacopla al frontend de realizar filtros de fecha complejos.

⚡ Indexación de Búsquedas (Database Indexes)

Campo Único (Unique)

`email` y `username` indexados para búsquedas rápidas de login y asegurar que no haya duplicación de cuentas.

Compuestos (Compound)

`{ disponible: 1, categoria: 1 }` en Menú y `{ fecha_reserva: 1, estado: 1 }` en Reservas. Optimizan búsquedas en base a múltiples filtros.

Índice TTL (Time-To-Live)

En Auditoría, el campo `fecha` expira automáticamente a los 30 días (`expires: 2592000`), autolimpiando el disco de logs obsoletos.

Triggers y Pipelines de Agregación

Hooks de Ciclo de Vida como Triggers (Mongoose Middleware):

  • Hook post('save') en Reserva:
    • Ejecuta un incremento atómico `$inc` en la colección `clientes` sumando 1 al campo `total_reservas_historicas`.
    • Inserta de manera transaccional un documento en `auditoria_logs` detallando la creación.
  • Hook post('findOneAndUpdate') en Reserva:
    • Detecta actualizaciones de estado (confirmación o cancelación) y genera el log correspondiente de forma automática.
Pipeline de Agregación para Ocupación por Hora (dashboardController.js) 
Reserva.aggregate([
  { 
    $match: { 
      fecha_reserva: { $gte: startOfDay, $lte: endOfDay }, 
      estado: { $ne: 'cancelada' } 
    } 
  },
  { 
    $group: { 
      _id: '$hora_reserva', 
      total_personas: { $sum: '$cantidad_personas' },
      total_reservas: { $sum: 1 }
    } 
  },
  { $sort: { _id: 1 } }
]);

Agrupa las reservas activas de un día, suma el volumen de personas por rango horario y ordena los resultados cronológicamente para renderizar los gráficos de la aplicación web.

Seguridad y Defensa del Sistema

🛡️ Inmunidad a SQL Injection

MongoDB es inmune a las inyecciones SQL tradicionales debido a que no utiliza un parser de texto plano para evaluar consultas, sino que estructura los comandos en BSON.

Bloqueo de NoSQL Injection: Aunque un atacante inyecte operadores lógicos como `{ "$ne": "" }` en el login, el sistema está protegido porque la validación criptográfica de contraseñas se hace en la memoria del backend mediante bcrypt.compare(), imposibilitando el bypass de clave.

Otros Mecanismos de Defensa Implementados:

  • Cifrado Unidireccional: Las contraseñas de los usuarios nunca se guardan en texto plano en la base de datos; se hashean con salting mediante bcryptjs.
  • Autorización Stateless (JWT): Los endpoints del panel de control están protegidos por tokens firmados criptográficamente. Si el token está alterado o caducó, el middleware de Express lo rebota al instante.
  • Desacoplamiento CORS: El servidor de Express restringe los métodos HTTP aceptados y las cabeceras origen permitidas para evitar la inyección de peticiones cruzadas maliciosas.

Despliegue y Flujo de Trabajo (CI/CD)

Arquitectura Distribuida en Producción

  • Base de Datos (MongoDB Atlas): Aloja los datos en la nube. Requiere Whitelisting de IP (`0.0.0.0/0`) para permitir conexiones seguras desde los servidores de Railway y Vercel.
  • Backend (Railway): Despliega automáticamente la carpeta `backend` conectada a GitHub, exponiendo una URL segura mediante HTTPS y configurando variables sensibles de forma aislada.
  • Frontend (Vercel): Aloja la carpeta `frontend` (React + Vite) de forma estática en CDNs distribuidos, apuntando a la URL del backend mediante la variable `VITE_API_URL`.

🚀 Integración Continua (CI/CD)

Cero procesos manuales de despliegue:

Push a GitHub Master ➔ Despliegue en Caliente en Vercel & Railway ➔ Actualización en Producción en 30s.

Las configuraciones de redirección en Vercel se manejan mediante el archivo vercel.json en la raíz del frontend para evitar errores de refresco en la SPA.
